## 1. 도메인 개요

### 1.1 도메인 목적

PDF 도메인은 ModuMark의 차별화 기능인 **무료 PDF 처리 통합**을 담당한다. 마크다운과 PDF를 단일 앱에서 처리함으로써, 별도 도구(알PDF, Adobe)를 쓰던 사용자가 ModuMark 하나로 모든 문서 작업을 완결할 수 있게 한다.

**Phase 1 구현 현황**: PDF 뷰어·병합·분할·마크다운→PDF 변환이 완료되었으며, 원래 Phase 3 예정이었던 썸네일 미리보기(dnd-kit 정렬), 페이지 추출, 다중 페이지 선택, Undo 30단계 기능까지 Phase 1에서 조기 구현되었다. 기존 `/pdf/merge`, `/pdf/split` 별도 라우트 대신 `/pdf` 통합 에디터 라우트(`PdfEditor.tsx`)로 아키텍처가 통합되었다.

**Phase 2A 구현 현황**: OCR(Tesseract.js, 한국어 지원), PDF 압축(3단계 레벨), DPR 고해상도 렌더링, 뷰어 전용 모드(모드 탭), OCR 언어팩 IndexedDB 캐싱이 완료되었다.

**Phase 3 방향**: 모드 탭(edit/viewer/compress/ocr)을 완전히 제거하고 **뷰어 단일 화면**으로 전환한다. 사용자는 PDF를 열면 즉시 뷰어에서 내용을 확인하고, 뷰어 안에 배치된 액션 버튼(분할·압축·OCR·병합)을 통해 다이얼로그 오버레이 방식으로 원하는 작업을 수행한다. 모드 전환 없이 모든 작업이 단일 화면에서 완결된다. 여러 PDF를 동시에 열 경우 상단에 파일 탭(브라우저 탭 방식)으로 표시되며, 각 파일은 독립 컨텍스트를 유지한다. 병합 흐름은 "PDF 열기 → 파일 추가 버튼 → 추가 파일 페이지가 뷰어 하단에 붙음 → 병합 저장"의 순서로 단순화된다.

### 1.2 연결된 비즈니스 문제

| 전체 BRD 문제 ID | 문제 | PDF 도메인 연관성 |
|-----------------|------|-----------------|
| P1 | 마크다운 에디터와 PDF 편집 도구 분리 | **직접 해결**: 동일 앱 내 PDF 처리 |
| P3 | PDF 편집·OCR·변환 도구가 마크다운과 미연동 | **직접 해결**: 마크다운↔PDF 상호 변환 |
| P4 | PDF 뷰어와 병합 기능이 별도 UI로 분리되어 사용자가 두 기능 사이를 오가며 작업해야 하는 불편함 | **직접 해결**: 단일 사이드바에서 열린 파일 네비게이션과 병합 대상 파일 목록을 통합 |

| 사무직 (박지현) | PDF 병합·분할, 무료 PDF 도구 대체 | 알PDF 수준의 병합·분할 기능 |

## 4. 핵심 비즈니스 규칙

| 규칙 ID | 규칙 | 근거 |
|---------|------|------|
| P-BR1 | 처리되는 PDF 파일은 어떠한 경우에도 외부 서버로 전송되거나 저장되지 않는다. OCR 포함 모든 PDF 처리는 클라이언트(브라우저 또는 데스크탑 앱) 내에서만 수행된다 | 로컬 우선 원칙, 개인정보보호법, 기업 보안 정책 준수 (보안 페르소나: 최성호) |
| P-BR2 | 서버리스 처리 시 단일 파일 크기는 5MB 이하로 제한한다 | Vercel 무료 티어 제약 (BRD 8.1) |
| P-BR3 | 병합 대상 파일은 최대 20개, 총 합산 100MB 이하로 제한한다 | 메모리·성능 안정성 |
| P-BR4 | OCR 처리는 우선적으로 클라이언트(Tesseract.js)에서 수행한다 | 서버 비용 최소화, 개인정보 보호 |
| P-BR5 | OCR 결과 신뢰도가 80% 미만인 경우 LOW_CONFIDENCE 경고를 표시한다 | 사용자에게 품질 정보 제공 |
| P-BR6 | PDF 변환 결과는 사용자의 로컬 파일 시스템에만 저장한다 | 로컬 우선 원칙 |
| P-BR7 | PDF 압축 기능을 제공한다. pdf-lib 이미지 압축 + 폰트 서브셋 방식으로 파일 크기를 줄이며, 모든 처리는 클라이언트에서 수행한다 | 알PDF 대체 완성도 향상, PROPOSAL-002 채택 (Phase 2) |
| P-BR8 | PDF를 열면 뷰어 단일 화면이 표시되며, 분할·압축·OCR·병합 등 모든 작업은 뷰어 안의 액션 버튼을 통해 다이얼로그 오버레이로 수행된다. 모드 탭은 존재하지 않으며 화면 전환 없이 모든 작업이 완결된다 | 뷰어 단일 화면 UX 원칙 (Phase 3) |
| P-BR9 | 병합은 "PDF 열기 → 파일 추가 버튼 → 추가 파일 페이지가 뷰어 하단에 순서대로 붙음 → 병합 저장" 흐름으로 수행된다. 사용자는 최종 병합 결과를 저장하기 전에 페이지 순서를 뷰어에서 직접 확인할 수 있다 | 병합 흐름 단순화 및 결과 예측 가능성 확보 (Phase 3) |
| P-BR10 | 여러 PDF를 동시에 열면 상단에 파일 탭(브라우저 탭 방식)으로 표시된다. 각 파일 탭은 독립적인 컨텍스트(페이지 위치, 진행 중인 작업 상태 등)를 가지며, 탭 전환 시 해당 파일의 마지막 상태가 복원된다 | 다중 PDF 동시 작업 지원 (Phase 3) |
| P-BR11 | 좌측 사이드바는 단일 파일 네비게이션 공간으로 사용된다. 사이드바 상단에는 현재 열린 PDF의 페이지 썸네일 목록(뷰어 네비게이션)이 표시되고, 병합 모드에서 파일을 추가하면 해당 파일의 페이지들이 사이드바 하단에 이어서 표시된다. 뷰어 네비게이션과 병합 파일 목록은 분리된 패널이 아닌 연속된 단일 사이드바 내에 통합되어, 사용자는 화면 전환 없이 파일 내용 확인과 병합 대상 관리를 동시에 수행할 수 있다 | 뷰어↔병합 통합 워크플로우 단순화 (P-G6), 사이드바 이탈 클릭 제거 (Phase 3) |

## 5. 성공 지표 (KPI)

| KPI | 정의 | 목표 |
|-----|------|------|
| MD→PDF 변환 성공률 | 변환 요청 대비 성공 비율 | 99% 이상 |
| PDF 병합 성공률 | 병합 요청 대비 성공 비율 | 99% 이상 |
| OCR 평균 정확도 | 한국어 표준 문서 OCR 인식률 | 80% 이상 |
| PDF 처리 완료 시간 | 10MB 이하 파일 처리 완료까지 소요 시간 | 10초 이내 P90 |
| PDF 기능 주간 사용 횟수 | 주간 PDF 기능 사용 횟수 합계 | 1,000회 이상 (출시 3개월 후) |
| 통합 사이드바 사용률 | 병합 기능 사용 시 통합 사이드바를 통해 파일을 추가하는 비율 | 병합 작업의 80% 이상이 통합 사이드바 경유 (Phase 3 출시 1개월 후) |
| 뷰어↔병합 전환 클릭 수 | 뷰어에서 병합 작업 시작까지 필요한 추가 클릭 수 | 0회 (사이드바 이탈 없이 병합 대상 추가 가능) |

## 6. 범위 정의

### 6.1 포함 범위 (In-Scope)

| 기능 | 설명 |
|------|------|
| PDF 뷰어 | PDF 파일 렌더링, 페이지 이동, 줌 |
| PDF 병합 | 여러 PDF를 하나로 합치기 (최대 20개) |
| PDF 분할 | 특정 페이지 범위 추출하여 새 PDF 생성 |
| 마크다운 → PDF | 현재 편집 중인 마크다운 문서를 PDF로 변환 |
| PDF OCR | 스캔·이미지 PDF에서 텍스트 추출 (Tesseract.js) |
| PDF 압축 | PDF 파일 크기 압축 (이미지 압축 + 폰트 서브셋, Phase 2A 완료) |
| PDF → 마크다운 | PDF 텍스트 레이어를 마크다운으로 변환 |
| 진행률 표시 | 처리 중인 작업의 진행 상태 표시 |
| 뷰어 단일 화면 | 모드 탭 없이 PDF를 열면 즉시 뷰어가 표시되며, 모든 작업은 뷰어 내 액션 버튼과 다이얼로그 오버레이로 수행 (Phase 3) |
| 파일 탭 기반 다중 PDF 관리 | 여러 PDF를 동시에 열면 상단 파일 탭으로 관리. 각 탭은 독립 컨텍스트 유지 (Phase 3) |
| 액션 버튼 툴바 | 뷰어 내 분할·압축·OCR·병합 액션 버튼 제공. 각 기능은 다이얼로그 오버레이로 실행 (Phase 3) |
| 통합 사이드바 | 좌측 단일 사이드바에서 뷰어 페이지 네비게이션(상단)과 병합 추가 파일 목록(하단)을 통합 표시. 병합 대상 파일 추가 시 사이드바 하단에 해당 파일 페이지가 연속 표시됨 (Phase 3) |

### 6.2 제외 범위 (Out-of-Scope)

| 제외 항목 | 이유 |
|----------|------|
| PDF 직접 편집 (텍스트 수정) | 구현 복잡도 과다, PDF 구조 특성상 어려움 |
| PDF 전자서명 | 법적 요구사항 복잡 |
| PDF 양식(Form) 작성 | MVP 범위 초과 |
| PDF 암호 설정/해제 | Phase 2 이후 검토 |
| 5MB 초과 파일 서버 처리 | Vercel 무료 티어 제약 |
| 모드 탭 UI (edit/viewer/compress/ocr 전환) | Phase 3에서 완전 제거. 뷰어 단일 화면으로 대체 |
