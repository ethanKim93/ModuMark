## 1. 도메인 개요

### 1.1 도메인 목적

Platform 도메인은 **Windows 네이티브 앱과 웹 브라우저라는 두 가지 실행 환경을 추상화**하여 Editor/PDF 도메인이 플랫폼에 독립적으로 동작할 수 있게 한다. SEO 최적화된 웹 서비스 기반과 데스크탑 앱 배포를 담당한다.

### 1.2 연결된 비즈니스 문제

| 전체 BRD 문제 ID | 문제 | Platform 도메인 연관성 |
|-----------------|------|----------------------|
| P4 | 데스크탑 전용 도구의 접근성 제한 | **직접 해결**: 웹 + 데스크탑 동시 지원 |
| P1 | 여러 도구 간 컨텍스트 전환 | .md 파일 기본 앱 등록으로 단일 앱 워크플로우 구현 |

## 3. 이해관계자 (도메인 레벨)

| 이해관계자 | 관심사 | 요구사항 |
|-----------|--------|---------|
| 웹 사용자 (이수진) | 설치 없이 즉시 사용 | 빠른 초기 로딩, 모바일 반응형 |
| 데스크탑 사용자 (김민준) | .md/.pdf 파일 연결, 오프라인 사용 | 파일 연결 등록 (.md → 마크다운 에디터, .pdf → PDF 뷰어), 오프라인 동작 |
| 보안 민감 기업 사용자 (최성호) | 로컬 우선 아키텍처로 기업 보안 정책 준수 | 파일 전송 없는 서버리스/클라이언트 처리 구조, 데스크탑 앱 오프라인 동작 |
| Google (검색 엔진) | 크롤링 가능한 구조 | SSR/SSG, 시맨틱 HTML, sitemap |
| Google AdSense | 트래픽 품질, 콘텐츠 충분성 | 충분한 텍스트 콘텐츠 페이지 |
| Google AdSense 정책 팀 | 콘텐츠 충분성, 네비게이션 일관성 | 모든 공개 페이지에서 일관된 헤더/푸터 네비게이션 구조 제공. 어떤 페이지에서도 서비스 전체를 탐색할 수 있는 구조 |

## 4. 핵심 비즈니스 규칙

| 규칙 ID | 규칙 | 근거 |
|---------|------|------|
| PL-BR1 | 웹 공개 페이지는 SSR 또는 SSG로 렌더링된다 | SEO 검색 엔진 크롤링 필수, AdSense 승인 조건 |
| PL-BR2 | 모든 공개 페이지는 고유한 title과 meta description을 가진다 | SEO 최적화, CTR 개선 |
| PL-BR3 | sitemap.xml과 robots.txt를 자동 생성하여 제공한다 | 검색 엔진 인덱싱 최적화 |
| PL-BR4 | Windows 배포 파일은 코드 서명(Code Signing)을 적용한다 | SmartScreen 경고 방지, 사용자 신뢰 확보 |
| PL-BR5 | `.md 및 .pdf` 파일 기본 앱으로 ModuMark를 등록할 수 있다 | 기존 Typora 사용자 워크플로우 동일 경험, PDF 뷰어로서의 단일 앱 경험 |
| PL-BR11 | `.pdf` 파일을 더블클릭 시 ModuMark PDF 뷰어에서 열린다 | PDF 파일 연결을 통해 외부 PDF 뷰어 없이 ModuMark 내에서 즉시 열람 가능 |
| PL-BR6 | 앱 업데이트는 사용자 동의 후 설치된다 | 사용자 통제권 보장 |
| PL-BR7 | 웹과 데스크탑에서 동일한 핵심 기능을 제공한다 | 크로스 플랫폼 일관된 경험 |
| PL-BR8 | 모든 파일 처리(편집, PDF 변환, OCR 등)는 클라이언트에서 수행되며, 파일 데이터가 외부 서버로 전송되지 않는다 | 로컬 우선 아키텍처, 기업 보안 정책 준수, 개인정보보호법 (보안 페르소나: 최성호) |
| PL-BR9 | "보안 안전한 로컬 우선" 아키텍처를 제품 소개 페이지에 명시적으로 안내한다 | 보안 민감 사용자 대상 차별점 커뮤니케이션 |
| PL-BR10 | 모든 페이지(랜딩 포함)에서 다크/라이트/시스템 테마 전환을 제공한다 | 첫 방문 사용자도 즉시 선호 테마로 사용 가능, UX 일관성 |
| PL-BR12 | 랜딩 페이지에서 데스크탑 앱을 바로 다운로드할 수 있어야 한다 | 앱 인지 → 다운로드 전환율 극대화. Hero CTA 및 네비게이션에서 즉시 접근 가능 |
| PL-BR13 | /download 전용 페이지를 제공한다 | OS별(.exe/.msi) 구분된 다운로드 버튼, 버전 정보(버전명·릴리즈 날짜·파일 크기), 설치 안내를 한 페이지에서 제공. SEO 크롤링 대상 공개 페이지로 관리 |
| PL-BR14 | /download 페이지는 sitemap.xml에 포함된다 | 검색 엔진이 다운로드 페이지를 인덱싱할 수 있어야 함. "마크다운 편집기 다운로드" 키워드 SEO 가치 확보 |
| PL-BR15 | GitHub Releases URL은 환경 변수 또는 상수 파일에서 중앙 관리한다 | 릴리즈 URL이 분산되면 버전 업 시 다중 파일 수정 필요. 단일 위치에서 관리하여 유지보수 비용 최소화 |
| PL-BR16 | 모든 공개 페이지는 동일한 SiteHeader/SiteFooter 컴포넌트를 사용한다 | 네비게이션 일관성. Google AdSense 승인 요건인 "사이트 전반의 일관된 탐색 구조" 충족 |
| PL-BR17 | SiteHeader는 로고, 주요 페이지 네비게이션(마크다운 에디터·PDF 도구·가이드·About), ThemeToggle, 모바일 햄버거 메뉴를 포함한다 | 어떤 공개 페이지에서도 서비스 핵심 기능으로 즉시 접근 가능. 모바일(375px) 사용성 보장 |
| PL-BR18 | SiteFooter는 도구 링크, 법적 페이지 링크, 연락처(이메일: modu.markdown@gmail.com, GitHub Issues) 정보를 포함한다 | 서비스 운영자 정보 및 지원 창구 명시. AdSense 승인 요건인 "연락처 정보 제공" 충족 |

## 6. 범위 정의

### 6.1 포함 범위 (In-Scope)

| 기능 | 설명 |
|------|------|
| Next.js 웹 앱 | SSR/SSG 기반 Next.js 16.1.6 App Router 웹 서비스 |
| SEO 기반 구조 | 메타태그, OG 태그, sitemap.xml, robots.txt, 구조화 데이터 |
| Tauri 데스크탑 앱 | Windows 네이티브 앱 (Tauri 2.0 + Next.js) |
| 파일 시스템 접근 | 로컬 .md/.pdf 파일 읽기/쓰기 (Tauri FS API) |
| .md/.pdf 파일 연결 | Windows에서 .md 파일 더블클릭 → `/markdown` 에디터, .pdf 파일 더블클릭 → `/pdf` 뷰어로 자동 라우팅 |
| 자동 업데이터 | 새 버전 감지 및 사용자 동의 후 업데이트 설치 |
| 창 상태 관리 | 창 크기·위치 저장/복원, 최대화/최소화 |
| 앱 설정 | 테마(라이트/다크/시스템), 언어(한국어) 설정 |
| 반응형 레이아웃 | 모바일(375px) ~ 데스크탑(1440px) 대응 |
| Vercel 배포 | 웹 앱 서버리스 배포 ✅ Phase 1 완료 |
| **테마 전환** | 랜딩 페이지 포함 전체 페이지에서 테마 전환 가능 (next-themes) ✅ Phase 1 완료, 랜딩 페이지 Phase 2A 보완 |
| **앱 다운로드 페이지 (/download)** | OS별 구분 다운로드(Windows .exe / .msi), 버전 정보(버전명·릴리즈 날짜·파일 크기), 설치 안내, sitemap.xml 포함 |
| **GitHub Releases 연동** | 릴리즈 URL 중앙 관리 (`lib/releases.ts` 상수 파일). 초기에는 하드코딩, 향후 GitHub API 자동 참조 구조로 확장 가능한 설계 |
| **SiteHeader/SiteFooter 통합 컴포넌트** | 모든 공개 페이지에서 공유하는 단일 헤더·푸터 컴포넌트. 로고·주요 네비게이션·ThemeToggle·모바일 햄버거 메뉴(SiteHeader) + 도구·법적·연락처 링크(SiteFooter) |

### 6.2 제외 범위 (Out-of-Scope)
